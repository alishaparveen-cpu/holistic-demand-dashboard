# Data refresh

The dashboard reads static `data_*.json` files. They're regenerated by one command and,
in production, by a daily GitHub Action.

## Manual refresh (local)

```bash
aws sso login --profile redshift-data        # Redshift access
source ~/.allo_google_ads.env                # Google Ads + GBP creds (chmod 600, gitignored)
bash scripts/refresh_all.sh                  # regenerates data, prints OK/FAIL per step
git add data_*.json && git commit -m "data refresh" && git push
```

`refresh_all.sh` is resilient — one failing source does not abort the rest, and every step
prints `OK` or `FAIL` with the error. It refreshes the **Redshift-** and **API-driven** files
(scorecard, phase-2, diagnostic, Google Ads city/daily/funnel, GMB insights).

> `data.json` (the core bookings funnel) still has manual Google-Sheet inputs per
> `DATA_SOURCES.md` and is **not** part of this automated refresh yet.

## Scheduled refresh (GitHub Action)

`.github/workflows/refresh-data.yml` runs daily (01:30 UTC ≈ 07:00 IST) and on manual
dispatch, then commits the refreshed data. **It only works once you add the repo secrets**
(Settings → Secrets and variables → Actions → New repository secret) — I cannot add these
for you:

| Secret | What |
|--------|------|
| `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` | An IAM user with **Redshift Data API** access to the `warehouse` cluster. CI cannot use your SSO profile — it needs static keys. |
| `GOOGLE_ADS_DEVELOPER_TOKEN`, `GOOGLE_ADS_CLIENT_ID`, `GOOGLE_ADS_CLIENT_SECRET`, `GOOGLE_ADS_REFRESH_TOKEN` | Google Ads API |
| `GBP_CLIENT_ID`, `GBP_CLIENT_SECRET`, `GBP_REFRESH_TOKEN` | Google Business Profile (GMB insights) |

⚠️ **Rotate** any credential that was shared over chat before putting it in repo secrets.
